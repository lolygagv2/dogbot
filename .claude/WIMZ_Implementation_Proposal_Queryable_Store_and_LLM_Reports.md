# WIM-Z Implementation Proposal: Queryable Closed-Loop Store + Session-Report LLM Layer

Owner: Morgan Hill (morgan@wimzai.com)
Status: Proposal / build order. Hand to the App, Robot (Edge), and Relay Claude Code instances.
Contract: This document **defers to `WIMZ_Data_Architecture_Spec.md` v0.1** for all schema. It sequences the build and adds one new table. It does not redesign anything in the spec. Where this proposal needs a new table, the spec is bumped first (per spec rule 9), then built.

---

## 0. Why this exists

Current state: the device records video only when a user in AI mode runs a trick, and everything else lands in an error/event log. The data is captured but not queryable, not linked, and not yet useful for statistics or for the moat. A few SQL cleanups on event data are already done.

The spec already designs the fix. The moat is the closed loop (cue, response, reward, outcome over time), not raw events. This proposal turns the existing log into the spec's linked `training_attempt` and `outcome_snapshot` rows, then adds a cheap natural-language session report on top.

Two hard cost rules, because we are pre-funding:

- **No always-on or conversational LLM in production.** Deferred until after the raise.
- **LLM is a thin, per-call API layer on the Relay only.** It generates a session report from already-structured rows. It costs cents per session and touches no hot path on the robot.

---

## 1. Scope

In scope:

- A. Edge: make the event log conform to the spec core tables and assemble `training_attempt` rows. Backfill existing logs. Stop discarding hard negatives.
- B. Relay: generate a per-session natural-language report via the Claude API from structured rows. Store it. No always-on model.
- C. App: display the session report and surface the human-review queue.

Out of scope (deferred to post-funding):

- Always-on or conversational on-device LLM.
- Full cloud corpus build beyond the `landing/` stage in the spec.
- Real-time streaming inference changes.

---

## 2. Schema change required first (Relay/shared)

The session report needs a home. Add one additive table to `WIMZ_Data_Architecture_Spec.md`, bump it to v0.2, then build. Additive only, no change to existing tables.

```sql
-- Natural-language summary of one session, generated by the Relay LLM layer.
CREATE TABLE session_report (
  report_id     TEXT PRIMARY KEY,          -- UUIDv7, generated on Relay
  session_id    TEXT NOT NULL,             -- FK -> session.session_id
  dog_id        TEXT NOT NULL,             -- denormalized for fast read
  generated_at  INTEGER NOT NULL,          -- epoch ms, UTC
  model_id      TEXT NOT NULL,             -- e.g. 'claude-haiku-4-5'
  input_hash    TEXT NOT NULL,             -- sha256 of the structured input, for idempotency
  summary_text  TEXT NOT NULL,             -- the report shown in the app
  stats_json    TEXT,                      -- the raw numbers the summary was built from
  created_at    INTEGER NOT NULL,
  updated_at    INTEGER NOT NULL
);
```

`model_id` is mandatory, same provenance discipline as machine labels. `input_hash` makes regeneration idempotent: same session data in, no duplicate report.

---

## 3. Workstream A: Edge / Robot (highest priority)

Goal: every training interaction becomes a linked `training_attempt`, and the existing history is backfilled. This is the work that turns "a log" into "a moat."

A1. **Conform the event writer to the spec.** Every machine event writes `event_type`, `ts`, `model_id`, `confidence`, `payload` per spec section 5. No anonymous labels. Keep `pose_rejected` and low-confidence detections (spec rule 5) instead of dropping them.

A2. **Assemble `training_attempt` on every trick.** When an AI-mode trick runs, write the linked unit described in spec section 6: `cue_issued` event, `pose` event, `media_asset` row for the clip, `dispense_log` + `treat_dispensed`, then the `training_attempt` row tying them together (`trick_label`, `cue_event_id`, `response_event_id`, `latency_ms`, `success`, `reward_dispensed`, `dispense_id`, `media_id`).

A3. **Compute `outcome_snapshot` on a schedule.** Per dog, per `trick_label`, roll up `success_rate` and `avg_latency_ms` over the latest window. This is what makes "is the dog improving" answerable in one query.

A4. **Backfill.** Run the one-time migration in spec section 6: parse existing `time + trick label` filenames into `media_asset` + `training_attempt` rows (`label_source='machine'`, `success` unknown unless a sidecar says otherwise), and turn existing bark/pose text logs into `event` rows. Do not lose history. It is the start of the longitudinal record.

A5. **Write discipline.** Batched transactions per attempt or every few seconds (spec section 7) to protect the SD card. Media on the filesystem, DB stores pointers (spec rule 1).

Acceptance: a single SQL query returns a dog's sit-success trajectory over time (spec section 8 example query runs and returns real rows), and every new trick produces a complete `training_attempt`.

---

## 4. Workstream B: Relay / session-report LLM layer

Goal: turn one session's structured rows into a short, owner-friendly report. Cheap, per call, no always-on model.

B1. **Trigger.** On session-close sync (preferred) or on explicit app request. Never on the robot hot path.

B2. **Input assembly (deterministic, no LLM).** For the `session_id`, pull: the `training_attempt` rows, the relevant `outcome_snapshot` deltas (this session vs the prior window), and a count of notable `event` rows (errors, `pose_rejected`). Build a small structured `stats_json`. Compute `input_hash`.

B3. **Idempotency.** If a `session_report` already exists for this `session_id` with the same `input_hash`, return it. Do not re-call the API.

B4. **Generation.** One Claude API call. Recommended model `claude-haiku-4-5` for cost (this is a short, structured summarization task, not reasoning-heavy). System prompt instructs: factual, owner-friendly, no invented numbers, use only the provided stats, short. Pass `stats_json` as the input.

B5. **Store.** Write the `session_report` row with `model_id`, `generated_at`, `input_hash`, `summary_text`, `stats_json`.

B6. **Cost posture.** Per-session Haiku call on a few hundred tokens is a fraction of a cent. No persistent model, no GPU, no recurring infra. This is the entire LLM footprint until after the raise.

Acceptance: closing a session produces one stored `session_report`; re-syncing the same session does not generate a second API call; the summary contains only numbers present in `stats_json`.

Example target output: "Elsa, 12-minute training session. 6 sit attempts, 5 successful, average response 2.1s. Sit success is up from 60% last week to 83%. One pose was rejected on low light near the window. Overall, steady improvement on sit."

---

## 5. Workstream C: App

C1. **Display the session report.** Read `session_report` for a session and show `summary_text` with the underlying `stats_json` available on tap.

C2. **Human-review queue.** Surface the highest-value unlabeled clips for confirmation (spec section 8 query: acted-on, mid-confidence attempts with media). A human confirmation writes `label_source='human'`, which is the highest-value row in the system. This is the flywheel.

C3. **No new data authority.** The app stays authoritative only for human-entered fields per spec section 2. It does not originate behavioral data.

Acceptance: a user can read a plain-language summary of the last session and confirm or correct at least the flagged clips, with corrections persisting as human-sourced labels.

---

## 6. Sequencing

1. Schema bump to v0.2 (add `session_report`). Shared, blocks B.
2. Workstream A (Edge): A1, A2, A3 first; A4 backfill once A2 is stable.
3. Workstream B (Relay): after A2 produces real `training_attempt` rows to summarize.
4. Workstream C (App): after B writes its first reports.

A is the priority. Without linked attempts there is nothing to report on and no moat to accumulate. B and C are thin once A is real.

---

## 7. Explicit non-goals for this pass

- No conversational assistant.
- No always-on inference beyond current detection.
- No corpus training run. Sync stays at `landing/` per spec until post-funding.
- No schema changes beyond the single additive `session_report` table.
